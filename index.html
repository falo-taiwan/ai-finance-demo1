🔍 專案背景與統一入口說明

本專案旨在對決並比較三款頂尖 AI 模型在爬取公開資訊觀測站（MOPS）申報異動公告（CAE Tracker）時的策略效能。 該目標網站（https://howtofightfraud.com/cae-tracker/）設有單次查詢顯示上限 88 筆記錄的硬性物理限制。 若查詢範圍內的公告數大於等於 88 筆，系統將截斷輸出並提示「已達顯示上限」，導致一般爬蟲遺漏資料。

📌 挑戰痛點

網站總資料量 pool 達 13,239 筆，直接按年度查詢會嚴重觸發 88 筆限制。爬蟲必須採用更細緻的二維或三維切片分格（年、月、市場），但在取得完整資料的同時，必須將 HTTP 請求次數降到最低以避免被伺服器封鎖。

⚡ 策略差異

Claude 採用年→市場→月自適應分格；ChatGPT 採用預讀 ID 清單的智慧名單格網；而 Gemini 採用年→月→市場動態自適應分格。本平台對其效能進行了量化對決。

🌐 三方資料融合 (Data Fusion)

由於各家爬蟲因分格演算法的不同與網路回應抖動，單獨執行時均存在微幅的覆蓋率遺漏。本儀表板透過主金鑰去重技術，將三家資料合併為 100% 完整公眾可檢索資料集（11,005 筆），並提供統一搜尋下載。

Claude Strategy

Claude: 自適應層級分格法 (Adaptive Hierarchical Slicing)

年 (32) → 市場 (4) → 月 (12)

優點： 優先進行大顆粒度（年度）請求。若單次請求結果筆數小於 88，則直接收錄，不展開子節點。只有在單次查詢「撞上限」時，才分裂成市場，若市場再撞，才分裂成月份。這確保了稀疏年代（如民國 80 年代）只需 1 次請求即可完成，毋需發送 48 次請求。

缺點： 由於分裂是「年 → 市場 → 月」，當某個年度的「上市」市場仍然撞 88 上限時，它會將該市場下的 12 個月份全部展開。在資料密集的近年（民國 100~114 年），幾乎所有月份都必須被展開，這導致發送了大量冗餘請求（共計 704 次）。

Gemini Strategy

Gemini: 動態時序自適應分格法 (Dynamic Timeline Slicing)

年 (32) → 月 (12) → 市場 (4)

優點： 由於絕大多數年度都會撞 88 上限，Gemini 在第二層分裂時選擇了「時序月份」而非「市場」。一個年度的單月資料大於 88 筆的機率極低。因此在第二層展開為 12 個月後，幾乎所有的月份均未達 88 筆上限，不需要進入第三層（市場分裂）。

結果： 只有 12 個月份（主要集中在民國 103~105 年的特定繁重月份）需要分裂為市場。這使得 Gemini 的總 HTTP 請求數銳減至 368 次（比 Claude 的 704 次節省了將近一半的伺服器負載與時間），而覆蓋率卻達到了完美的 11,005 筆，堪稱最優雅、最高效的演算法。

ChatGPT Strategy

ChatGPT Pro: 智慧快取與名單重建法 (Smart Index Cache Lookup)

ChatGPT 採用了一種非常「工程師思維」的巧妙方法：它首先存取了公開的後台 API 端點 /wp-json/mops-audit/v1/record-ids。這個 API 端點會吐出整張資料表的公告 ID 清單（包括公司、日期和序號）。

透過這份 ID 名單，ChatGPT 在本地端先計算出哪些年月、哪些市場的公告數量會大於 88 筆。隨後，它只針對會撞上限的年月進行市場細分查詢，對於安全的年月則只發送一次年月查詢。這種「開圖外掛」式的策略讓其請求數壓到最低的 357 次，並能精確得知哪些資料被截斷。
